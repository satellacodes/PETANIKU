## üìù Issue Creation Guidelines

### Before Creating an Issue:

1. **Search existing issues** - Avoid duplicates
2. **Check documentation** - Solution might already exist
3. **Test in different environments** - Confirm it's not environment-specific
4. **Gather necessary information** - Logs, screenshots, steps to reproduce

### Required Information:

- **Clear title** - Descriptive and specific
- **Detailed description** - What, when, where, how
- **Environment details** - Version, OS, browser
- **Steps to reproduce** - Exact, numbered steps
- **Expected vs actual behavior** - Clear comparison

### Quality Standards:

‚úÖ **Good Issues:**

- Specific and detailed
- Includes reproduction steps
- Provides necessary context
- Respectful and constructive

‚ùå **Poor Issues:**

- Vague or general complaints
- Missing reproduction steps
- Demanding or entitled tone
- Not related to project scope

### Issue Labels:

- `bug` - For software defects
- `enhancement` - For improvements
- `documentation` - For docs issues
- `question` - For questions
- `help wanted` - Community help needed
- `good first issue` - Beginner-friendly

### Response Time:

- **Acknowledgement:** Within 24-48 hours
- **Triage:** Within 3-5 business days
- **Update:** Weekly if issue is active

## üîß For Developers

### When Working on Issues:

1. **Claim the issue** - Comment that you're working on it
2. **Ask questions** - If anything is unclear
3. **Provide updates** - Regular progress reports
4. **Link PR to issue** - Use "Closes #issue-number"

### Issue States:

- `open` - New or being discussed
- `assigned` - Someone is working on it
- `in progress` - Actively being worked on
- `review` - Ready for review
- `closed` - Resolved or completed

### Priority Levels:

- **P0 (Critical):** Security issues, data loss, system down
- **P1 (High):** Major functionality broken
- **P2 (Medium):** Important features not working optimally
- **P3 (Low):** Minor issues, enhancements, cosmetic

## üìä Issue Management Process

### 1. **Submission**

- User submits issue using template
- Automated label assignment based on content
- Initial review within 24 hours

### 2. **Triage**

- Verify completeness of information
- Assign priority and labels
- Determine if it's a valid issue
- Assign to appropriate milestone

### 3. **Discussion**

- Community input welcomed
- Technical discussion
- Solution brainstorming
- Scope definition

### 4. **Implementation**

- Assign to developer
- Development and testing
- Code review
- Merge to appropriate branch

### 5. **Verification**

- Testing of fix/feature
- User acceptance testing
- Documentation updates
- Close issue when verified
